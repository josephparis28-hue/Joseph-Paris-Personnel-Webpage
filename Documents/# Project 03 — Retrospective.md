## 1. What Went Well

### **Clear Project Direction**

The revised scope statement provided a strong foundation for the entire project. Defining the purpose, deliverables and constraints early made the development process smoother and focused. The clarity of the scope helped prevent scope-creep which kept the project aligned with the final requirements.

### **Consistent Application of PMUPM Principles**

The planning phase effictively followed the PMUPM framework. Creating the WBS, schedule and TAME risk analysis gave structure to the work. This made it easier to track progress. Tasks broken into smaller components helped maintain momentum to avoid feeling overwhelmed.

### **Improved Technical Execution**

The two‑page website demonstrated growth from previous projects. The HTML structure was more semantic, the CSS was more organized and the overall design was more cohesive. The unified stylesheet ensured consistency across both pages. The optional enhancements added polish without accessibility compromise.

### **Professional Repository Organization**

The GitHub repository was clean, well‑structured and easy to navigate. The README.md clearly explained the project, linked to the live site and connected to the documentation folder. This created a professional presentation suitable for a true portfolio.

### **Time Management**

The project stayed close to the planned schedule. Completing the documentation first made the development phase more efficient, and testing early helped catch issues before deployment.

---

## 2. What Went Wrong

**Initial Scope Misalignment**
The first version of the scope statement focused on a single‑page website. This did not fully match the final project requirements till revision later in the process. Although it was corrected, it caused a slight delay and highlighted the importance of aligning the scope with the rubric from the beginning.

**CSS Complexity**
Styling both pages with a single stylesheet introduced challenges. Some styles conflicted or required refactoring to maintain consistency across entire website. This added extra time to the styling phase and required reorganizing the CSS for clarity. There many times I had to trouble shoot why something broke when a new feature was added.

**Feature Overload Temptation**
There was a temptation to add too many optional enhancements (animations, modals, toggles). While these features improve the site, they risked distracting from the core requirements. It took discipline to prioritize required deliverables before enhancements.

**Testing Gaps**
Early testing missed a few layout issues on smaller screens. These were fixed later, but it showed that testing should have been integrated earlier and more frequently throughout development.

---

## 3. What I Would Do Differently

**Start With the Rubric First**
In future projects, I will begin by mapping the rubric directly to the scope statement before writing anything else. This ensures perfect alignment from the start and avoids rework.

**Create a More Modular CSS Structure**
Using a more modular approach (e.g., separating typography, layout and components) do make the stylesheet easier to maintain. This reduces conflicts and speeds up future styling work.

**Integrate Testing Into Each Phase**
Instead of testing at the end, I would test after completing each major section (header, hero, about, etc.). This will catch issues earlier and reduce the amount of cleanup required at the end.

**Limit Enhancements Until Core Requirements Are Complete**
While enhancements add value, they should come only after the required structure and styling are fully complete. In future projects, I will lock enhancements behind a core-complete milestone to avoid distraction.

**Document Decisions as I Go**
Keeping brief notes during development can make the retrospective even stronger. Documenting decisions, challenges and solutions in real time will provide richer insights for reflection.
